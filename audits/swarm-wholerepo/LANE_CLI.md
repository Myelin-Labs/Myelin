# Myelin Swarm Audit — CLI Surface

> Verifier-only review. No fixes proposed. Scope: `cli/src/main.rs` (17,553
> lines), `cli/Cargo.toml`, plus the two new rehearsal shell scripts
> `scripts/myelin_public_testnet_rehearsal_prepare.sh` (264 lines) and
> `scripts/myelin_public_testnet_rehearsal_live.sh` (286 lines). Cross-references
> to `MYELIN_PRODUCTION_GATE.md`, `MYELIN_PRODUCTION_REHEARSAL_REPORT.md`,
> `MYELIN_SESSION_L2_PLAN.md`, `docs/public-testnet-rehearsal-runbook.md`,
> the templates under `docs/templates/public-testnet-rehearsal/`, and the
> `cellscript/examples/myelin/*.cell` fixtures.
>
> The branch under review is `arthur/session-l2-production-evidence-fixes`,
> ~4,924-line diff vs `main` (CLI side). The CLI is the only crate whose
> surface was materially changed for production evidence; `state/`, `consensus/`,
> `mempool/` were audited separately and are referenced only at the boundary.

## Verdict

**Conditional PASS for the documented fixture-backed graph; do NOT merge for
public-testnet rehearsal use.** The branch adds an internally consistent
production-evidence machinery on top of the existing CLI: signed external DA
receipts, threshold-lock deployment evidence, authority signature evidence,
court-economics deployment evidence, operator custody / runbook documents, and
final-script preflight checks. Every new command has a schema check, a parser,
a verifier, and a binding to the manifest/package it claims to extend, and
all field-level checks against the existing court-bundle / DA-manifest /
settlement-intent invariants hold in the static fixture path. The branch is
**not** safe for public-testnet rehearsal yet because the gate does not run
the new evidence commands directly — the `production-evidence-complete`
posture is asserted through `verify_session_submission_readiness` only, and
the `da-availability-schema/mode/commitment/ready` checks in
`MYELIN_PRODUCTION_GATE.sh:1183-1211` reject any non-mock receipt (the gate
expects `external_receipt_count == 0` and `production_ready == false`), so
the gate and the rehearsal scripts disagree about whether a signed
production-SLA receipt is allowed in the default run. Several smaller
defects (signature coverage gaps, fixture keys hard-coded inline, doc/CLI
field drift, error taxonomy collapsing into one exit code, the production
gate's missing rehearsal-script hook, and a deterministic dependence on
in-memory ordering in the `attestation_hashes` / `attester_pubkey_hashes` /
`signatures` arrays) need to be resolved before the rehearsal scripts are
treated as more than a paper trail.

Top risks, in order:

1. **Production-gate and rehearsal scripts disagree on whether a signed
   production receipt can be exercised in the default gate.** The gate
   asserts `external_receipt_count == 0`, `external_receipt_checked == False`,
   `external_receipt is None`, and `production_ready is False` on
   `availability`; the rehearsal scripts produce exactly the opposite shape
   for the public-testnet artifact. Either the gate should call
   `session_external_da_receipt` + `session_da_manifest --external-da-receipt`
   on the rehearsal receipt, or the readiness claim is asserted in a path the
   gate never reaches.
2. **`session_external_da_receipt` signs only the typed fields; the
   `receipt_commitment` covers `receipt_hash` (a hash of the raw JSON bytes)
   separately, but the availability commitment folds in `receipt_hash`,
   `receipt_commitment`, `provider_signature_verified`, and
   `production_guarantee_checked` rather than the typed fields directly.** A
   receipt with `receipt_bytes` mutated post-signature fails `receipt_hash`
   but passes every typed-field check; a receipt with the typed fields
   mutated post-signature fails the provider signature but the receipt JSON
   can be re-serialized around it and `receipt_hash` will look valid because
   the `receipt_id`/`availability_window` fields that feed
   `receipt_commitment` are not part of the provider signature domain. The
   `receipt_commitment` is a redundant cross-check on the typed values only;
   it is not signature-covered.
3. **All four production-ready paths collapse into a single exit code.** The
   CLI only has four `CliError` variants (`Io`, `Json`, `Consensus`,
   `InvalidFixture`), and `main()` exits `1` for any of them
   (`cli/src/main.rs:977-994`). The branch's commit messages claim "missing
   evidence / bad evidence / stale evidence" are distinguished; the code
   surfaces them as the same `InvalidFixture` message string in stderr. A
   rehearsal operator cannot tell from `$?` why the gate failed.
4. **Hard-coded fixture keys embedded in `da_availability_evidence`,
   `settlement_authority_authentication`, the unit test fixtures, and the
   `external_da_receipt` fixture.** These keys are exercised for every
   `session da-manifest` and `session settlement-package` invocation even on
   real public-testnet inputs; the only way to use real keys is to construct
   the receipt or authority file externally and feed it in. The fixture
   `committee_id = "myelin-replicated-da-committee-testnet-beta-v1"` is
   permanently named `testnet-beta-v1` regardless of the network.
5. **The new evidence helpers have no dedicated verifier that round-trips
   back to the manifest.** `verify_session_da_manifest` recomputes
   `da_availability_evidence(...)` and asserts the manifest's copy is
   `==`-equal (line 6398), but the verdict is in a JSON `checks[].ok` field
   and there is no machine-readable separate proof that the receipt bytes
   match the receipt_hash. The same comment applies to
   `session_court_economics_deployment_evidence` — the helper builds a
   commitment, but `verify_settlement_package` only re-derives
   `court_economics_commitment`; the `evidence_commitment` is asserted
   `==`-equal at `cli/src/main.rs:4274-4277, 4327-4330` but the
   `audit_report_hash` field is never compared to anything.

### 2. Signature/scheme coverage

The branch introduces six new signature domains:

1. `myelin:external-da-receipt-provider-signature:v2` — `blake3_chunks(domain, [schema, provider, namespace, payload_hash, segment_root, receipt_id, availability_window, service_level, retention_seconds, retrieval_endpoint, audit_log_commitment])` (cli/src/main.rs:3022-3036). Secp256k1 over this 32-byte hash with the recovered pubkey hash truncated to 20 bytes (4032-4037). **Missing `receipt_id` and `availability_window`** (F-CLI-02).
2. `myelin:session-da-availability-attestation-message:v1` — `blake3_chunks(domain, [session_id, court_bundle_hash, payload_hash, segment_root, proof_molecule_hash, committee_id, member])` (3457-3467). Secp256k1 over this 32-byte hash. **Hard-coded attester set** (F-CLI-05).
3. `myelin:session-settlement-authority-cell-auth:v1` — `blake3_chunks(domain, [authority_data_hash, session_id, participant_set_hash, escrow_input_cells_hash, session_lineage_commitment])` (3672-3676). Secp256k1 over this 32-byte hash with two fixture signers (3685). **Hard-coded signer set** (F-CLI-06).
4. `myelin:session-authority-signature-evidence:v1` (commitment) — `blake3(hasher, [schema, signature_scheme, participant_set_hash, threshold_le, signer_count_le, message_hash, signer_pubkey_hashes[], signatures[], attestation_hashes[], [u8;1] from signature_verified])` (3983-4001). Commitment only — the per-signer secp256k1 signatures are verified in the loop (3944-3948).
5. `myelin:session-settlement-authority-cell-signature-attestation:v1` — `blake3_chunks(domain, [message_hash, participant_set_hash, pubkey_hash, signature])` (3949-3952). Per-signer attestation.
6. `myelin:session-da-availability-signature-attestation:v1` — `blake3_chunks(domain, [attestation_message, pubkey_hash, signature_bytes])` (3480-3483). Per-attester attestation.

Field coverage summary:

| Domain | Fields covered | Fields NOT covered | Risk |
|---|---|---|---|
| `external-da-receipt-provider-signature:v2` | schema, provider, namespace, payload_hash, segment_root, service_level, retention_seconds, retrieval_endpoint, audit_log_commitment | **receipt_id, availability_window** | F-CLI-02 — receipt can be re-emitted with different `receipt_id` and the same signature still verifies |
| `session-da-availability-attestation-message:v1` | session_id, court_bundle_hash, payload_hash, segment_root, proof_molecule_hash, committee_id, member | (nothing obvious) | OK; attesters are fixture keys (F-CLI-05) |
| `session-settlement-authority-cell-auth:v1` | authority_data_hash, session_id, participant_set_hash, escrow_input_cells_hash, session_lineage_commitment | (nothing obvious) | OK; signers are fixture keys (F-CLI-06) |
| `session-authority-signature-evidence:v1` (commitment) | full evidence schema fields | (commitment covers all fields) | OK |
| `session-settlement-authority-cell-signature-attestation:v1` | message_hash, participant_set_hash, pubkey_hash, signature | (per-signer attestation; does not include session_id, escrow, etc.) | OK as an attestation; the message_hash covers the broader set |

Substitution risks:

- **Field substitution**: `receipt_id` is not in the signature domain. An attacker who holds a valid signature for `(payload_hash, segment_root, service_level=production, retention=30d, https_endpoint, audit_log_commitment)` can swap the `receipt_id` and re-emit the receipt with a different `availability_window` (e.g. `production-retention-90d` to claim a longer retention). The signature verifies; the verifier does not check the window string against any external reference. (F-CLI-02)
- **Missing field**: if the receipt JSON omits `audit_log_commitment` entirely (the field is optional), the producer-side `optional_string_field` returns `Ok(None)` (line 2897) and the message hash includes `""` (line 3034). The signature verifies; the `production_guarantee_checked` returns `false` because `audit_log_commitment.is_some_and(...)` (line 3408) is false. But the receipt is still accepted by `external_da_receipt_evidence` (line 2982-3002) — only the downstream `production_ready` boolean changes. (Not a defect, but worth noting that the verifier does not reject the receipt itself, only the production claim.)
- **Empty string**: caught by `optional_string_field` at line 2886-2887 (`must be non-empty when present`).
- **Overlong string**: not explicitly bounded; `external_da_receipt_provider_message_hash` only feeds the field as a `&[u8]` slice into `blake3_chunks`. blake3 hashes any length deterministically. So an overlong `availability_window` (e.g. 1 MiB) is accepted. The cost is memory, not security.
- **Byte ordering / encoding determinism**: `blake3_chunks` is deterministic; secp256k1 signatures are non-deterministic (RFC 6979 deterministic nonces are not used by the `secp256k1` crate by default for recoverable signatures), but the **recovery** of the pubkey from the signature is deterministic. The `signature_verified` flag and the `provider_signature_verified` flag (line 2950-2951) compare the recovered pubkey hash to the `provider_pubkey_hash`, which is deterministic. The signature bytes themselves may vary between two valid signatures, but the `provider_message_hash` is deterministic. The `availability_commitment` does not include the signature bytes, only the pubkey hash and the message hash, so the commitment is deterministic.

### 3. Production-SLA enforcement

The diff binds `service_level = "production"`, `retention_seconds >= 30d`,
`https://` retrieval_endpoint, 32-byte `audit_log_commitment` in
`external_da_receipt_production_guarantee_checked`
(`cli/src/main.rs:3398-3409`). The check is called from:

- `da_availability_evidence` line 2976-2981 (during receipt construction)
- `da_availability_evidence` line 3523 (`production_ready = testnet_beta_ready && external_receipt.as_ref().is_some_and(|receipt| receipt.production_guarantee_checked)`)
- `da_availability_production_ready` line 3411-3418 (re-derived in verifier)
- `verify_session_da_manifest` line 6415 (`production_ready == da_availability_production_ready(&manifest.availability)`)

Consistency across callsites:

- The check is the same function (`external_da_receipt_production_guarantee_checked`) called from all four sites.
- The function is `pure` (no I/O, no external state).
- The function's `MIN_PRODUCTION_RETENTION_SECONDS = 30 * 24 * 60 * 60` (line 3404) matches `MYELIN_PRODUCTION_REHEARSAL_REPORT.md:36` ("30 days") and matches the runbook example (`docs/public-testnet-rehearsal-runbook.md:184` uses `--retention-seconds 2592000`, which is exactly 30 days).

Doc claim vs code reality:

| Doc claim | Code reality |
|---|---|
| `retention_seconds >= 30 days` | `>= 30 * 24 * 60 * 60` (line 3406) — equal. |
| `https://` retrieval_endpoint | `endpoint.starts_with("https://") && endpoint.len() > "https://".len()` (line 3407) — accepts `https://x` (8 chars). Does not check that the URL parses, that the host is reachable, or that the certificate is valid. |
| 32-byte audit_log_commitment | `commitment.len() == 64 && hex::decode(commitment).is_ok()` (line 3408). **Any 32-byte hex string passes.** |

Signature coverage of SLA fields:

- `provider_message_hash` includes `service_level`, `retention_seconds`, `retrieval_endpoint`, `audit_log_commitment` (lines 3031-3034). So a signature is bound to these fields.
- But `receipt_id` and `availability_window` are NOT in the signature domain (F-CLI-02).
- And `receipt_hash` (which is the `blake3` of the raw receipt bytes) is not signature-covered in a useful way — a receipt with mutated bytes fails `receipt_hash` but the `provider_message_hash` was computed over the typed fields, so the signature still verifies against the original typed fields. The verifier does not cross-check that `receipt_hash` equals `blake3(receipt_bytes)` where `receipt_bytes` is the JSON from which the typed fields were extracted. (F-CLI-03)

### 4. Error taxonomy

`CliError` has four variants (`cli/src/main.rs:977-985`):

| Variant | Source |
|---|---|
| `Io(#[from] std::io::Error)` | File system errors during JSON loading |
| `Json(#[from] serde_json::Error)` | JSON parse errors |
| `Consensus(#[from] myelin_consensus::ConsensusError)` | Finality engine errors |
| `InvalidFixture(String)` | All schema, signature, evidence, and binding errors — collapsed into one variant |

`main()` exits with `1` for any variant (`cli/src/main.rs:990-994`). The
production gate and the rehearsal scripts cannot distinguish "missing
evidence", "bad evidence", and "stale evidence" from `$?` alone. The
`MYELIN_PRODUCTION_REHEARSAL_REPORT.md:88-89` statement about "The CLI
should reject unreplaced cryptographic templates" is enforced inside
`normalize_*_evidence` (the placeholder hashes are still 32 bytes, so the
length check passes; the actual rejection comes from signature verification
or commitment mismatch), not from a separate exit code.

Differences the branch's commit messages claim:

- **Missing evidence**: in `external_da_receipt_evidence`, `None` is returned for the optional `receipt_path` (line 2845-2847). No error, just `Ok(None)`. The downstream `da_availability_evidence` is called with `external_receipt = None` (line 6168) and `production_ready = false`.
- **Bad evidence**: returned via `InvalidFixture("external DA receipt provider_signature does not match provider_pubkey_hash")` (line 2953) or `InvalidFixture("external DA receipt payload_hash mismatch: expected ...")` (line 2905).
- **Stale evidence**: not distinguished from bad evidence in code. A receipt that was valid yesterday but whose `payload_hash` no longer matches the manifest is treated the same as a receipt that was never valid. The error message says "mismatch" but does not say "stale".

The rehearsal scripts use `jq -e '.valid == true'` (e.g.
`scripts/myelin_public_testnet_rehearsal_prepare.sh:48-50, 101`) to check
verifier output, but if the CLI errors before emitting JSON, the jq check
fails with a confusing error message.

## Open questions

1. **F-CLI-01 (CRITICAL)**: Does the production gate need to run the new evidence commands directly, or does the readiness report cover the gate's claim? Currently the gate reads mock-only reports and asserts `external_receipt_count == 0`, but the rehearsal scripts produce non-mock receipts. Either path is defensible; the branch needs to commit to one.
2. **F-CLI-02 (HIGH)**: Should `receipt_id` and `availability_window` be added to the signature domain? If the provider signs the same `payload_hash` / `segment_root` twice and emits two different `receipt_id`s, both signatures are valid but only one receipt is canonical.
3. **F-CLI-04 (HIGH)**: Should the CLI surface distinct error variants for "missing evidence", "bad evidence", and "stale evidence"? The shell scripts work around the lack by parsing `jq '.valid == true'`, but `myelin` itself cannot be scripted by exit code.
4. **F-CLI-05 (HIGH)**: Should the `da_availability_evidence` attester set be loaded from a config file or environment variable, or is the fixture committee intentional (because the rehearsal is documented as mock-backed)?
5. **F-CLI-06 (HIGH)**: Should `settlement_authority_authentication` use the operator-supplied signer set instead of the `[0x11, 0x22]` fixture? The current design forces the operator to read the generated package and feed signatures back via `authority-signature-evidence`, which is correct but adds a step.
6. **F-CLI-07 (HIGH)**: Should `bare_hex_*_arg` and `parse_hex_*` consistently accept or reject `0x`-prefixed hex? Today they both reject (because `decode_hex_bytes` strips `0x` but `parse_hex_32` does not), but a future caller might trip over the inconsistency.
7. **F-CLI-10 (MEDIUM)**: The "testnet-beta ready" invariant requires `ckb_enforceable_checked`, which the runbook only documents for `network = "ckb-mainnet"`. Is the runbook's Phase 3 example (`--network ckb-testnet` + `--ckb-enforceable-checked --testnet-beta-ready`) meant to be a testnet rehearsal of the mainnet invariant?
8. **F-CLI-13 (MEDIUM)**: The default `MYELIN_DA_AUDIT_LOG_COMMITMENT = "0x$(hex_repeat a5 32)"` in `prepare.sh` line 62 will be rejected by the CLI's `bare_hex_32_arg`. Does the rehearsal script currently work, or does this script need to strip the prefix?
9. **F-CLI-15 (MEDIUM)**: Should the rehearsal receipt JSON record the `provenance` of the secret-key path, so downstream consumers can tell a real-provider receipt from a CLI-helper receipt?
10. **F-CLI-17 (MEDIUM)**: Should `verify_session_da_manifest` expose `availability_commitment_algorithm` as a separate `checks[]` entry?
11. **F-CLI-21 (LOW)**: Should there be a unit test asserting the literal `availability_commitment_algorithm` string?
12. **F-CLI-25 (LOW)**: Should the `da-availability-ready` check decompose into separate `provider-signature-verified` and `production-sla-checked` sub-checks?
13. **F-CLI-29 (LOW)**: Should the readiness verifier recompute `expected_ckb_tx_hash` from the underlying submission JSON instead of trusting the `expected_ckb_tx_hash` string in the context report?
14. **F-CLI-34 (LOW)**: Is 20-byte blake3 truncation sufficient for `pubkey_hash`, given that the same 20 bytes also feed `ckb_lock_args_hash`?
